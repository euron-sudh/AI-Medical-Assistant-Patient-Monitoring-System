1 Welcome & Project Overview

▼

What is MedAssist AI?

MedAssist AI is a real-time, agentic AI-powered medical assistant platform designed for hospitals, telemedicine providers, and health startups. It combines symptom analysis, medical report interpretation, continuous patient monitoring, voice-based interaction, and intelligent alerting into a unified system. The platform leverages 7 specialized AI agents that collaborate autonomously to deliver accurate, context-aware medical insights to both patients and healthcare professionals.

Why Are We Building This?

Healthcare systems around the world face critical challenges:

  • Delayed diagnosis: Patients wait days or weeks for specialist appointments. Our symptom analysis agent provides immediate, evidence-based preliminary assessments.
  • Information overload: Doctors spend hours reading lab reports. Our Report Reader agent extracts and highlights critical values in seconds.
  • Missed deterioration: Hospital patients deteriorate unnoticed. Our monitoring agent watches vitals 24/7 and predicts deterioration 30-60 minutes ahead.
  • Drug interactions: Medication errors are a leading cause of preventable harm. Our drug interaction agent catches conflicts in real-time.
  • Access barriers: Rural and underserved communities lack specialist access. Telemedicine and voice-based AI bridge that gap.
  • Care fragmentation: Treatment plans get lost between visits. AI-generated care plans with adherence tracking keep patients on track.

How a Request Flows Through the System

Let's trace what happens when a patient submits a symptom check:

  1. Patient types "I have chest pain and shortness of breath" in the chat UI (Next.js frontend)
  2. Frontend sends POST to /api/v1/symptoms/session/:id/message via the API client (Axios with JWT header)
  3. Request hits the API Gateway (Nginx), which validates the rate limit and forwards to Flask
  4. Flask's auth middleware validates the JWT token and extracts the user identity
  5. The symptoms route receives the request and calls the symptom service
  6. The service invokes the Agent Orchestrator, which classifies the intent as "symptom analysis"
  7. Orchestrator dispatches to the Symptom Analyst Agent (GPT-4o) and the Triage Agent (GPT-4o-mini) in parallel
  8. Symptom Agent uses tools: search_medical_knowledge_base (Pinecone RAG), query_patient_history (PostgreSQL)
  9. Agent generates differential diagnosis, urgency score, and recommended action
  10. Triage Agent assigns ESI Level 2 (high urgency for chest pain)
  11. Results saved to symptom_sessions table in PostgreSQL
  12. Because ESI is 1-2, a Celery task fires to alert the on-call physician via push notification and SMS
  13. Response streams back to the frontend via WebSocket (chat:stream event) for a real-time typing effect
  14. HIPAA audit middleware logs the entire interaction to audit_logs

The Service Layer Pattern

Why separate Routes, Services, and Models?

This is the most important architectural pattern in the backend. It keeps code organized, testable, and maintainable.

Routes (in api/v1/) handle HTTP concerns: parse request data, validate input, return HTTP responses. They should be thin — just a few lines.

Services (in services/) contain business logic: "what should happen when a patient books an appointment?" Rules, validations, orchestration, error handling. This is where the real work happens.

Models (in models/) represent database tables as Python classes. They define the schema and provide query methods. No business logic here.

How AI Agents Work

Our AI layer uses an orchestrator pattern. A central orchestrator receives every AI request and routes it to the right specialist agent(s). Think of it like a hospital reception desk that directs patients to the right department.

  1. Orchestrator receives the request and classifies intent using GPT-4o with function calling
  2. It selects one or more specialist agents based on the intent (can run in parallel)
  3. Each agent has its own system prompt, tools, and memory
  4. Agents use function calling to invoke tools (search knowledge base, query patient history, etc.)
  5. Short-term context is stored in Redis (conversation buffer) and long-term knowledge in Pinecone (vector embeddings)
  6. The orchestrator aggregates responses and returns a unified result

Agent 1: Symptom Analyst Agent

Model: GPT-4o Owner: Chidi Henry Priority: High

What it does: Conducts multi-turn symptom interviews, builds differential diagnosis lists, assigns urgency scores, and recommends next steps. It considers patient history, age, sex, and pre-existing conditions.

Tools:

  • search_medical_knowledge_base — RAG over medical literature embeddings in Pinecone
  • query_patient_history — fetch past diagnoses, allergies, medications from PostgreSQL
  • calculate_urgency_score — rule-based + ML urgency scoring
  • generate_differential_diagnosis — structured output of possible conditions with confidence scores
  • recommend_specialist — maps symptoms to specialist type

When triggered: Patient starts a symptom check via chat, voice, or body map

Example: Patient says "I've had a headache for 3 days, mostly behind my eyes, and it gets worse when I look at screens." The agent asks follow-up questions about vision changes, nausea, and stress levels, then generates a differential including tension headache, migraine, and eye strain, with urgency score 3/10.

Agent 2: Medical Report Reader Agent

Model: GPT-4o Vision Owner: Chidi Henry Priority: High

What it does: Ingests and interprets medical reports (lab results, imaging, pathology). Extracts structured data, flags abnormalities, provides plain-language explanations for patients, and generates trend analysis.

Tools:

When triggered: Patient or doctor uploads a medical report (PDF/image)

Example: Patient uploads a CBC lab report. Agent extracts all values, flags low hemoglobin (10.2 g/dL, normal 12-16), explains "Your hemoglobin is below normal range, which may indicate anemia. Your doctor may want to investigate the cause."

Model: GPT-4o-mini (fast) Owner: Chidi Henry Priority: Critical

When triggered: Automatically runs alongside Symptom Analyst. Also called for quick triage assessments.

6 How Production-Grade Projects Work

▼

What is a Monorepo vs Polyrepo?

A monorepo stores all project code in a single Git repository. Our project is a monorepo — frontend, backend, infrastructure, and docs all live together. This makes it easier to coordinate changes across the stack (e.g., if an API changes, the frontend that calls it is in the same repo).

A polyrepo splits each service into its own repository. Better for very large organizations with independent teams, but adds coordination overhead.

What is Docker and Why We Use Containers?

Docker packages an application and all its dependencies into a container — a lightweight, portable unit that runs the same everywhere. When you run docker-compose up, it starts 8 services (Postgres, Redis, InfluxDB, Elasticsearch, MinIO, Flask, Celery, Next.js) with the exact same versions and configurations that everyone else uses.

Without Docker, every developer would spend hours installing PostgreSQL 16, Redis 7, InfluxDB, and Elasticsearch with the right versions and configurations. "Works on my machine" would be a daily problem.

What is CI/CD?

CI (Continuous Integration) means every time you push code, automated tests run. If they fail, you know before merging. Our .github/workflows/ci.yml runs lint + tests on every Pull Request.

CD (Continuous Deployment) means once code is merged, it automatically deploys. Merge to develop → deploys to staging. Merge to main → deploys to production.

What is a Database Migration?

A migration is a version-controlled change to the database schema. Instead of manually running SQL on the production database (dangerous!), we write migration scripts that Alembic manages. Each migration has an upgrade (apply) and downgrade (revert) function. This means:

  • Every schema change is tracked in Git
  • We can reproduce the exact database state on any environment
  • We can roll back if something goes wrong
  • Multiple developers can make schema changes without conflicts

What is JWT Authentication?

JWT (JSON Web Token) is a compact, URL-safe token that contains user identity and permissions. When you log in, the server creates a JWT signed with a secret key. On every subsequent request, you send this token in the Authorization header. The server verifies the signature without querying the database.

We use two tokens: an access token (expires in 15 minutes, used for every API call) and a refresh token (expires in 7 days, used to get a new access token). This limits damage if a token is stolen.

What is HIPAA?

HIPAA (Health Insurance Portability and Accountability Act) is a US federal law that protects patient health information (PHI). Any application that handles medical data must comply. Key requirements:

  • Encryption at rest: All patient data encrypted in the database (AES-256)
  • Encryption in transit: All communication over TLS 1.3
  • Access control: Only authorized users can access patient data (RBAC)
  • Audit logging: Every access to PHI is logged (who, what, when, from where)
  • Data de-identification: Patient data sent to OpenAI has PII stripped
  • Minimum necessary: Users only see the minimum data needed for their role

What are WebSockets?

Normal HTTP is request-response: the client asks, the server answers, the connection closes. WebSockets maintain a persistent connection so the server can push data to the client at any time.

We need this for: live vitals streaming (heart rate updates every second), real-time monitoring alerts (doctor needs instant notification), streaming AI responses (token-by-token chat), and push notifications.

What is a Task Queue (Celery)?

Some operations take too long for a synchronous API response: analyzing a medical report with GPT-4o Vision (5-10 seconds), sending notifications to hundreds of users, or processing batch vitals uploads from IoT devices.

Celery accepts these jobs and processes them in the background. The API returns immediately with "processing" status, and the client is notified via WebSocket when the job completes. Redis acts as the message broker between the API and Celery workers.

What is RAG (Retrieval Augmented Generation)?

GPT-4o is powerful but its training data has a cutoff date, and it doesn't know about our specific medical knowledge base. RAG solves this:

  1. Index: We convert medical literature (ICD-10 codes, drug interactions, symptom-disease mappings) into vector embeddings using text-embedding-3-large and store them in Pinecone
  2. Retrieve: When a patient asks about symptoms, we convert their query to a vector and find the most relevant medical knowledge from Pinecone
  3. Generate: We feed the retrieved context plus the patient's question to GPT-4o, which generates an answer grounded in our medical knowledge base

This ensures our AI agents use up-to-date, verified medical information rather than relying solely on the model's training data.

Unit, Integration, and E2E Tests

Unit tests test a single function or class in isolation. "Does the urgency scoring function return the right score for these inputs?" Fast, focused, many of them.

Integration tests test how components work together. "Does the vitals API correctly save to PostgreSQL and sync to InfluxDB?" Tests real database interactions.

E2E (End-to-End) tests simulate real user behavior. "Can a patient log in, start a symptom check, receive a diagnosis, and book an appointment?" Cypress drives a real browser through the full flow.

What is Infrastructure as Code (Terraform)?

Instead of clicking through the AWS console to create servers, databases, and networks, we define everything in .tf files. Run terraform apply and it creates the entire infrastructure. This means:

  • Infrastructure changes are reviewed in PRs just like code
  • We can recreate the entire environment from scratch in minutes
  • No "tribal knowledge" about how the server was configured

What is Kubernetes?

Kubernetes (K8s) manages Docker containers in production. It handles: auto-scaling (spin up more containers during traffic spikes), self-healing (restart crashed containers), rolling updates (deploy new versions without downtime), and load balancing (distribute requests across containers). We use AWS EKS (Elastic Kubernetes Service) as our managed K8s cluster.

Step-by-Step Push Workflow

  1. Pull latest main: git checkout main && git pull origin main
  2. Create your branch: git checkout -b feature/module/your-description
  3. Write your code. Commit often with proper messages.
  4. Push your branch: git push -u origin feature/module/your-description
  5. Open a Pull Request on GitHub targeting the main branch
  6. Add PK as reviewer (and your squad lead if applicable)
  7. Wait for approval + CI green, then Squash Merge
  8. Delete your branch after merge

PR Review Process

  • Every PR must target main
  • Title follows commit format: feat(module): description
  • Include: Task ID, what changed, how to test, screenshots (for frontend)
  • At least 1 approval required (PK or squad lead)
  • All CI checks must pass
  • Use Squash Merge (not regular merge or rebase)

Rules You Must Never Break

Rule A: NEVER push directly to main

Why: Direct pushes bypass code review and CI. A single bug could take down production or corrupt patient data. All code goes through PRs.

Rule B: NEVER merge your own PR without approval

Why: A second pair of eyes catches bugs, security issues, and architectural problems. Healthcare software requires extra scrutiny.

Rule C: NEVER put secrets, passwords, or API keys in code

Why: Git history is permanent. Even if you delete a secret later, it's in the commit history forever. Use .env files (gitignored) and AWS Secrets Manager.

Rule D: NEVER combine unrelated changes in one branch

Why: If one change needs to be reverted, you'd have to revert the other too. Each branch = one logical change.

Rule E: ALWAYS pull latest main before creating a new branch

Why: Starting from stale code means merge conflicts and integration issues later.

10 Common Patterns & Best Practices

▼

How to Add a New API Endpoint

  1. Define the route in app/api/v1/your_module.py. Use a Flask Blueprint. Keep it thin — parse input, call service, return response.
  2. Create/update the service in app/services/your_service.py. Put all business logic here. This is where validations, DB queries, and external calls happen.
  3. Create/update the model in app/models/your_model.py if you need a new database table or fields.
  4. Create a migration if the model changed: flask db migrate -m "add xyz column" then flask db upgrade.
  5. Register the blueprint in app/api/__init__.py if it's a new module.
  6. Write tests: unit test for the service, integration test for the endpoint.
  7. Update Swagger docs with docstrings in the route function (Flasgger reads them).
# Example: Adding a new endpoint

# 1. Route (app/api/v1/medications.py)
@medications_bp.route('/interaction-check', methods=['POST'])
@jwt_required()
def check_interactions():
    """Check drug interactions.
    ---
    tags: [Medications]
    parameters:
      - in: body
        schema:
          properties:
            drug_ids:
              type: array
    responses:
      200:
        description: Interaction results
    """
    data = request.get_json()
    result = MedicationService.check_interactions(
        drug_ids=data['drug_ids'],
        patient_id=get_jwt_identity()
    )
    return jsonify(result), 200

# 2. Service (app/services/medication_service.py)
class MedicationService:
    @staticmethod
    def check_interactions(drug_ids, patient_id):
        medications = Medication.query.filter(
            Medication.id.in_(drug_ids)
        ).all()
        # Call Drug Interaction Agent
        result = drug_interaction_agent.analyze(medications, patient_id)
        return result

How to Add a New Database Model

  1. Create a new file in app/models/ following the naming convention (snake_case.py).
  2. Define the class extending db.Model. Use UUID primary keys, add created_at and updated_at timestamps.
  3. Import the model in app/models/__init__.py so Alembic can find it.
  4. Generate migration: flask db migrate -m "add your_table"
  5. Review the generated migration in migrations/versions/ — Alembic sometimes guesses wrong.
  6. Apply: flask db upgrade

How to Add a New Frontend Page

  1. Create the page file in the correct route group: src/app/(patient)/your-feature/page.tsx
  2. Export a default React component. Use 'use client' directive if it needs interactivity.
  3. Create any needed components in src/components/your-feature/.
  4. Use React Query (useQuery) for data fetching, Zod for response validation.
  5. Use Tailwind CSS for styling. Follow existing page patterns.
  6. Add a basic render test in __tests__/pages/.

11 Glossary

▼
API
Application Programming Interface. A set of rules for how software components communicate. Our backend exposes a REST API that the frontend calls.
JWT
JSON Web Token. A compact, signed token containing user identity and permissions. Sent in every API request to prove who you are.
RBAC
Role-Based Access Control. Permissions are assigned to roles (patient, doctor, admin), not individual users. A doctor can see patient records; a patient cannot see other patients' records.
HIPAA
Health Insurance Portability and Accountability Act. US federal law requiring encryption, audit logging, and access controls for health data.
PHI
Protected Health Information. Any health data that can identify a patient: name, DOB, medical records, test results, etc.
ORM
Object Relational Mapper. Maps Python classes to database tables. SQLAlchemy is our ORM — write Python, it generates SQL.
REST
Representational State Transfer. API design pattern using HTTP methods: GET (read), POST (create), PUT (update), DELETE (remove).
WebSocket
Persistent bidirectional connection between client and server. Unlike HTTP (request-response), the server can push data anytime.
Docker
Platform for running applications in containers. A container packages the app + all dependencies into a portable, consistent unit.
K8s (Kubernetes)
Container orchestration platform. Manages scaling, self-healing, load balancing, and rolling updates for Docker containers in production.
CI/CD
Continuous Integration / Continuous Deployment. Automated testing on every push (CI) and automated deployment after merge (CD).
RAG
Retrieval Augmented Generation. Pattern where we retrieve relevant documents from a knowledge base and feed them to an LLM for grounded answers.
Vector DB
Database that stores and searches vector embeddings. Pinecone lets us find semantically similar medical knowledge using cosine similarity.
Embedding
A numerical representation (vector) of text that captures its meaning. Similar texts have similar embeddings, enabling semantic search.
Celery
Distributed task queue for Python. Accepts background jobs (report analysis, notifications) and processes them asynchronously.
Blueprint
Flask's way of organizing routes into modules. Each blueprint handles a section of the API (auth, patients, vitals, etc.).
Migration
Version-controlled database schema change. Managed by Alembic. Each migration has upgrade() and downgrade() functions.
SSR
Server-Side Rendering. The server generates HTML for each request. Faster initial load, better SEO. Next.js does this by default.
SSG
Static Site Generation. Pages are pre-built at build time. Fastest possible load, but content is static until rebuilt.
App Router
Next.js 14's routing system. File-based: a file at app/dashboard/page.tsx creates the route /dashboard.
ESI
Emergency Severity Index. 5-level triage system: 1 = immediate life-saving intervention, 5 = non-urgent. Our Triage Agent assigns these.
SOAP Notes
Clinical documentation format: Subjective (patient's complaint), Objective (exam findings), Assessment (diagnosis), Plan (treatment). Our Voice Agent generates these.
ICD-10
International Classification of Diseases, 10th Revision. Standardized codes for diagnoses. Example: J06.9 = "Acute upper respiratory infection."
LOINC
Logical Observation Identifiers Names and Codes. Standardized codes for lab tests. Example: 718-7 = "Hemoglobin [Mass/volume] in Blood."
NEWS2
National Early Warning Score 2. Scoring system that aggregates vitals into a single number predicting clinical deterioration risk.
MEWS
Modified Early Warning Score. Similar to NEWS2 but simpler. Both are calculated by our Monitoring Agent.
WebRTC
Web Real-Time Communication. Browser API for peer-to-peer video/audio calls. Daily.co handles the complexity for us.
MinIO
S3-compatible object storage that runs locally. We use it in development instead of AWS S3 for file storage.
Terraform
Infrastructure as Code tool. Defines cloud resources (servers, databases, networks) in .tf files. Reproducible, version-controlled infrastructure.
EKS
Elastic Kubernetes Service. AWS-managed Kubernetes. We don't manage the K8s control plane; AWS does.
Gunicorn
Production WSGI HTTP server for Python. Runs multiple Flask worker processes for concurrent request handling.
CORS
Cross-Origin Resource Sharing. Browser security policy. Our middleware allows the frontend (port 3000) to call the backend (port 5000).
